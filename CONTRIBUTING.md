# Contributing to LOL-GAI

**Repository:** https://github.com/Mikaayenson/lolgai  
**Site:** https://lolgai.io

Thank you for helping catalog GenAI binaries, abuse surfaces, and attribution signals.

## Source of truth

Each tool is **one YAML file** under `yaml/<vendor>/<tool-id>.yaml`.

- Put binaries, artifacts, signing, attribution signals, and `Commands` in that file.
- Do **not** hand-edit or commit `website/public/api/*` (generated by `bin/site.py` in CI; `make build` locally).
- Do **not** add vendor query languages (ES|QL, Sigma, KQL, etc.) or sample SIEM queries.

Validate before opening a PR:

```bash
python3 bin/validate.py
python3 bin/site.py
```

## Adding or updating a tool

1. Fork and create a branch.
2. Copy [YML-Template.yml](YML-Template.yml) or an existing file under `yaml/`. Field shapes are in the [schema](schema/lolgai.tool.schema.json). See [docs/verification-and-signing.md](docs/verification-and-signing.md) and [docs/reference-score.md](docs/reference-score.md).
3. Fill at minimum: `Name`, `Id`, `Category`, `Author`, `Details.Binaries`, at least one `Usage: normal` command, and ideally ranked `AttributionSignals`.
4. Tag commands with `Usage: normal` or `Usage: abuse`. Abuse is optional when there is no distinct dual-use CLI surface. Never label everyday invocation as abuse.
5. MITRE IDs belong on abuse recipes where applicable. Do not tag normal localhost `serve` with T1048; reserve exfil/network techniques for public bind (`0.0.0.0`) or outbound transfer.
6. Prefer accurate host artifacts and signing from official docs or lab observation. Set `UsableForAttribution: false` when signing is ad hoc, partial, or a shared host (VS Code). Paths: `~/…` on macOS/Linux; `%USERPROFILE%\\…`, `%LOCALAPPDATA%\\…`, or `%APPDATA%\\…` on Windows.
7. Set `VerificationLevel` honestly (`unverified` / `documented` / `observed`). Do not mark `observed` without evidence in `Coverage.EmpiricalNotes` or structured `Signing`.
8. `Uuid` is stable across Id renames (generate once with `python3 bin/backfill_tool_metadata.py`). Bump `Updated` (YYYY-MM-DD) on material edits.
9. When improving an entry, keep the original `Author` and add yourself under `Contributors` when useful.
10. `ReadinessGrade: A` requires strong attribution (usable signing on full/authenticode depth, or a high-reliability product-specific signal). Use B/C when recipes exist but attribution is weak. `bin/validate.py` enforces this.
11. Open a pull request. CI runs schema validation.

## Scope

| In scope | Out of scope |
|----------|----------------|
| Observable process / binary | IDE chat UI with no distinct process |
| Companion tunnels/helpers | Pure cloud SaaS, no host footprint |
| Install paths, configs, domains, signing | Generic `python` / `node` without GenAI entrypoint |
| Abuse recipes + attribution signals | Unverified malware samples (use other catalogs) |
| | Vendor-specific detection queries / rule packs |
